<objective>
Move `REFRESH MATERIALIZED VIEW CONCURRENTLY net_deposits_hourly` out of the request path and into a `tokio::spawn` background task that ticks every 60 seconds in `server.rs`.

Purpose: Every call to `get_net_deposits_from_view` currently blocks on a `REFRESH MATERIALIZED VIEW CONCURRENTLY` before querying. Under load, this serializes concurrent requests and adds latency. Refreshing on a background timer means requests always query a recently-refreshed (at most 60s stale) view instantly.

Output: Modified `reader.rs` (REFRESH removed) and modified `server.rs` (background task added). No migration needed — the required unique index `idx_net_deposits_hourly_asset_bucket` already exists.
</objective>

```rust
    // Background task: refresh net_deposits_hourly materialized view every 60 seconds.
    // Keeping this out of the request path means get_net_deposits_from_view queries instantly.
    {
        let mv_db = sui_pg_db::Db::for_read(database_url.clone(), db_arg.clone()).await?;
        tokio::spawn(async move {
            let mut interval = tokio::time::interval(std::time::Duration::from_secs(60));
            interval.set_missed_tick_behavior(tokio::time::MissedTickBehavior::Skip);
            loop {
                interval.tick().await;
                match mv_db.connect().await {
                    Ok(mut conn) => {
                        let _ = diesel::sql_query(
                            "REFRESH MATERIALIZED VIEW CONCURRENTLY net_deposits_hourly",
                        )
                        .execute(&mut conn)
                        .await;
                    }
                    Err(e) => {
                        tracing::warn!("MV refresh: failed to get DB connection: {}", e);
                    }
                }
            }
        });
        println!("Net deposits MV refresh task started (interval: 60s)");
    }
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Background task → PostgreSQL | `REFRESH MATERIALIZED VIEW CONCURRENTLY` runs with read-pool credentials |
| Request handler → MV query | Handler reads potentially 60s-stale view data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service | Background refresh loop | mitigate | `MissedTickBehavior::Skip` prevents tick pile-up if refresh takes >60s; connection errors are logged and swallowed, loop continues |
| T-02-02 | Information Disclosure | Stale MV data | accept | Up to 60s staleness is acceptable for net deposits reporting; documented in function doc comment |
| T-02-03 | Denial of Service | Extra DB connection pool | accept | One additional read pool connection for the MV refresh task; negligible pool pressure |
</threat_model>
