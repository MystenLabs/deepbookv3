Production runbook (zero-downtime index creation):
The Diesel migration runner wraps every migration in a transaction. `CREATE INDEX CONCURRENTLY`
CANNOT run inside a transaction and will fail immediately. Use `CREATE INDEX IF NOT EXISTS`
(non-concurrent) in the migration file. For zero-downtime production deployment:
  1. Connect to the production DB directly: `psql $DATABASE_URL`
  2. Run: `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_asset_supplied_sender ON asset_supplied(sender);`
  3. Run: `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_asset_withdrawn_sender ON asset_withdrawn(sender);`
  4. Then run the Diesel migration — `IF NOT EXISTS` makes both DDL statements no-ops.
  5. This avoids a table lock on production while still keeping migrations as the source of truth.

Note on `supplier` column LIKE queries: `get_asset_supplied` and `get_asset_withdrawn` filter on
`supplier` with a LIKE pattern (`to_pattern()` wraps the value with `%`). A leading wildcard
defeats a B-tree index. Do NOT add `idx_asset_supplied_supplier` — it would not be used by the
planner. The `supplier` column is a SupplierCap NFT object ID; those queries are low-frequency
admin/analytics calls, not the hot path.
</context>
