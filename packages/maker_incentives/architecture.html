Maker Incentives System

DeepBook V3 -- Permissionless Incentive Platform

Static layout showing what exists and where it lives. The scoring enclave runs on a Nitro-capable EC2 instance. The deepbook-server can run on the same instance (co-located) or on a separate EC2 — controlled by the --server-host flag. On-chain state lives on Sui. Scripts run from an operator machine.

EC2 Instance Detail

PostgreSQL Database

The central data store on the host. The deepbook-indexer continuously watches Sui chain events and writes order_updates, order_fills, and stake data into PostgreSQL tables. The deepbook-server reads from this same database to serve REST API queries.

deepbook-server (:9008)

Axum HTTP server that serves pool trading data over REST. The scoring enclave calls its /incentive_pool_data endpoint to get orders, fills, stakes, and pool coin metadata (decimals, symbols) for a given pool and time window.

deepbook-indexer

Long-running process that subscribes to Sui chain events and writes them into PostgreSQL. It populates the tables that deepbook-server reads from. Without the indexer, the database would have no data.

Networking: socat and vsock-proxy

Nitro Enclaves have no network access by default -- they can only communicate via VSOCK, a virtual socket protocol between the enclave and its host. We need two bridges:

  • socat (inbound): Operator scripts hit TCP port 3000 on the host. socat forwards that TCP traffic into the enclave over VSOCK CID:3000. This is how external callers reach the enclave.
  • vsock-proxy (outbound): The enclave needs to call deepbook-server. vsock-proxy listens on VSOCK port 9008 inside the enclave and forwards to DEEPBOOK_SERVER_HOST:DEEPBOOK_SERVER_PORT — wherever deepbook-server is running. Default: 127.0.0.1:9008 (co-located). For a separate instance, pass --server-host <private-ip> to the provisioning scripts.

Separate EC2 topology: The deepbook-server does not need to run on the same machine as the enclave. Since vsock-proxy runs on the host and makes the real TCP connection, it can reach any IP on the network. Just change the --server-host flag — no enclave rebuild needed, no PCR change, no re-registration. This means you don't need to Nitro-enable your existing deepbook-server instance.

deepbook-incentives (Enclave :3000)

Axum server running inside the Nitro Enclave TEE. It receives scoring requests, fetches pool data from deepbook-server (via VSOCK), runs compute_scores(), then BCS-encodes the results and signs them with an Ed25519 key that only exists inside the enclave. Four endpoints:

  • /process_data -- production: scores an epoch, returns signed BCS payload
  • /test_process_data -- dev only: same scoring, but the on-chain contract skips signature verification
  • /get_attestation -- returns the Nitro attestation document containing the enclave public key and PCR measurements
  • /health_check -- simple liveness probe
On-Chain Objects (Sui Move)

EnclaveConfig

A shared object that stores PCR values (Platform Configuration Registers). PCRs are SHA-384 hashes that uniquely identify the exact code running inside a Nitro Enclave -- PCR0 is the enclave image hash, PCR1 is the kernel hash, PCR2 is the application hash. When someone submits epoch results, the contract checks that the signing enclave's PCRs match the ones stored here.

  • Updated via update-pcrs.ts whenever the enclave image is rebuilt
  • In dev, set to all zeros with --debug to skip PCR verification
  • In production, must match the exact EIF that was built

Enclave MAKER_INCENTIVES

A shared object that stores the enclave's Ed25519 public key. This is the key the contract uses to verify that epoch results were actually signed by a legitimate enclave.

The public key is obtained by calling /get_attestation on the running enclave -- this returns a Nitro Attestation Document (a cryptographic proof from AWS) that contains the public key and proves it came from an enclave matching the PCRs in EnclaveConfig. The register-enclave.ts script fetches this attestation and submits it on-chain, where the contract extracts and stores the public key.

MAKER_INCENTIVES is a Move one-time-witness type that scopes this Enclave object to this specific package -- it prevents other packages from reusing or impersonating it.

IncentiveFund

Permissionless -- anyone can create one by calling create-fund.ts. The creator deposits DEEP tokens as the reward treasury and sets parameters for how rewards are distributed. One trading pool can have many funds targeting it (one-to-many), each with different params and funders. Rewards are always in DEEP -- the contract only accepts DEEP tokens for funding.

  • pool_id -- which DeepBook trading pool this fund incentivizes
  • reward_per_epoch -- DEEP distributed each epoch
  • alpha -- weight between size and spread in scoring
  • epoch_duration_ms, window_duration_ms -- timing params
  • min_spread, flow_floor -- scoring thresholds
  • treasury: Balance DEEP -- the pool of funds to distribute
  • submitted_epochs: Table -- tracks which epochs have been submitted, prevents the same epoch from being submitted twice for this fund

FundOwnerCap

A capability object returned to whoever calls create_fund(). Only the holder of this cap can:

  • Update reward_per_epoch via update_reward_per_epoch()
  • Update alpha via update_alpha()
  • Toggle the fund active/inactive via set_fund_active()

The cap is a Move key object owned by the creator's address. It can be transferred to a different address if ownership needs to change.

EpochRecord

Created by submit_epoch_results() -- one per fund per epoch. It is the output of the scoring process and stores:

  • The epoch time range (start_ms, end_ms)
  • Per-maker scores and DEEP reward allocations
  • claimed: Table -- tracks which makers have already claimed, preventing double-claims

EpochRecords are shared objects so any maker can read their allocation and call claim_reward() against it. Records are permanent -- unclaimed rewards never expire.

submitted_epochs (Table on IncentiveFund)

Not a separate object -- it is a field on IncentiveFund. It is a Move Table mapping epoch timestamps to booleans. When submit_epoch_results() is called, it checks this table first. If the epoch already exists, the transaction aborts. This prevents the same epoch from being scored and paid out twice (the "multiple submission exploit").

Operator Scripts

deploy.ts

What: Publishes the maker_incentives Move package to Sui.
When: Once, at initial setup. Again only if the contract itself needs upgrading.
Output: Creates deployed.<network>.json with packageId and enclaveConfigId.

  • --network testnet|mainnet -- target network

create-fund.ts

What: Creates a new IncentiveFund targeting a DeepBook pool and optionally deposits DEEP into it.
When: Whenever someone wants to start a new incentive program for a pool. Can be run multiple times for different pools or different params on the same pool.

  • --network -- target network
  • --pool-id 0x... -- the DeepBook pool to incentivize
  • --reward N -- DEEP per epoch
  • --fund N -- initial DEEP deposit into treasury
  • --alpha-bps N -- alpha in basis points (5000 = 0.5)
  • --epoch-duration-hours, --window-duration-hours

update-pcrs.ts

What: Sets the PCR values on the on-chain EnclaveConfig object.
When: After building/rebuilding the enclave EIF. The PCRs change every time the enclave image is rebuilt because the code hash changes.

  • --debug -- sets all PCRs to zeros (dev only, bypasses verification)
  • --pcr0 <hex> --pcr1 <hex> --pcr2 <hex> -- production PCR values from out/nitro.pcrs

register-enclave.ts

What: Fetches the attestation document from the running enclave and registers the enclave's public key on-chain.
When: After starting the enclave and after updating PCRs. Must be re-run whenever the enclave is restarted (new key pair).

  • --enclave-url http://host:3000 -- URL to reach the enclave (via socat)
  • --network -- target network

submit-epoch.ts

What: Submits scoring results for a single fund's epoch. POSTs to the enclave, gets back signed scores, builds a PTB to call submit_epoch_results on-chain.
When: Manual one-off submission. For automated daily runs, use submit-all-epochs.ts instead.

  • --fund-id 0x... -- which fund to score
  • --enclave-url -- enclave endpoint
  • --test -- use /test_process_data endpoint (dev)

submit-all-epochs.ts

What: Loops through ALL funds in deployed.<net>.json and submits the previous epoch for each. Idempotent -- checks is_epoch_submitted on-chain and skips already-done funds.
When: Daily, triggered by the systemd timer. This is the primary production script.

  • --network, --enclave-url
  • --dry-run -- hit the enclave, print scores, but skip the on-chain tx

backfill-epochs.ts

What: Replays missed epochs for a date range. For each day in the range, calls the enclave and submits on-chain. Skips epochs that were already submitted.
When: If the daily cron missed runs (server downtime, enclave crash, etc.).

  • --start-date YYYY-MM-DD
  • --end-date YYYY-MM-DD
  • --network, --enclave-url

upgrade-enclave.sh

What: Full enclave upgrade pipeline in one command: syncs code to EC2, terminates old enclave, rebuilds EIF, starts new enclave, restarts networking, updates PCRs on-chain, re-registers enclave.
When: Whenever scoring logic (scoring.rs, types.rs) changes.

  • --host -- EC2 IP
  • --key -- SSH key path
  • --network, --debug

The daily scoring and reward cycle. The submit-all-epochs.ts script is the orchestrator -- it reads the local config, calls the enclave for each fund, gets back signed scores, and submits the transaction on-chain. Makers claim rewards separately at their own pace.

Step 1: Trigger

The systemd timer fires daily at 00:15 UTC (15 minutes after the epoch boundary to ensure data is finalized). It launches submit-all-epochs.ts which reads deployed.<net>.json to get the list of all registered funds and their pool/param configs.

Step 2: For Each Fund

The script checks on-chain if this fund's epoch was already submitted (is_epoch_submitted). If not, it POSTs to the enclave's /process_data endpoint with pool_id, fund_id, and the fund's params (alpha, reward, epoch times).

Step 3: Enclave Scores

Inside the enclave, the server calls deepbook-server on port 9008 (via VSOCK proxy) to fetch order_updates, order_fills, stakes, and pool coin metadata for the epoch window. It runs compute_scores(), BCS-encodes the results, and Ed25519-signs them with the enclave's private key.

Step 4: Submit On-Chain

The script receives the signed BCS payload back from the enclave. It builds a Sui PTB calling submit_epoch_results() on the Move contract. The contract verifies the Ed25519 signature against the registered Enclave public key, debits the fund's treasury, and creates an EpochRecord with per-maker allocations.

Step 5: Results On-Chain

The EpochRecord is now a shared object on Sui containing each maker's score and DEEP allocation for this epoch. The IncentiveFund treasury has been debited by reward_per_epoch. The submitted_epochs table on the fund now marks this epoch as done.

Step 6: Makers Claim

At any time, a maker can call claim_reward() with their BalanceManager and the EpochRecord. The contract checks they haven't already claimed, looks up their allocation, and transfers DEEP to them. If multiple funds target the same pool, the maker can batch multiple claim_reward() calls into a single PTB.

Recovery

Missed Epoch: backfill-epochs.ts

If the daily cron missed runs (server was down, enclave crashed, etc.), this script replays epochs for a date range. It follows the same flow -- calls the enclave, submits on-chain -- but for historical epoch windows. Skips any that were already submitted.

  • --start-date YYYY-MM-DD --end-date YYYY-MM-DD
  • The systemd timer also has Persistent=true which catches missed runs on reboot

One-time setup to go from zero to a live system. Four phases: deploy the contract on Sui, build and start the enclave on EC2, register the enclave on-chain so the contract trusts it, then create an incentive fund and install automation. Each phase depends on the previous one.

Prerequisites (manual, before any scripts)

EC2 Instance

You need a Nitro-capable EC2 instance (e.g. m5.xlarge in us-east-2). Nitro Enclaves are an AWS feature — not all instance types support them. If starting from a fresh Amazon Linux instance, run setup-ec2.sh to install Docker, nitro-cli, vsock-proxy, socat, and configure the allocator. This is a separate EC2 from your deepbook-server — the enclave talks to deepbook-server over the network via --server-host.

deepbook-server Endpoints

The deepbook-server must expose the endpoints the enclave and indexer rely on. The enclave calls /incentive_pool_data/{pool_id}?start_ms=...&end_ms=... to fetch raw order, fill, and stake data for scoring. Additionally, the following 3 event query endpoints must be available (these were added to deepbook-server by a separate agent):

  • GET /maker_incentive_fund_created — query FundCreated events
  • GET /maker_incentive_epoch_results_submitted — query EpochResultsSubmitted events
  • GET /maker_incentive_reward_claimed — query RewardClaimed events

Verify these endpoints are working before proceeding. Without /incentive_pool_data, the enclave cannot score. Without the event endpoints, clients cannot query historical incentive activity.

Local Nautilus Repo

The Nautilus framework repo must be cloned locally at ~/projects/nautilus. Our scoring code (scoring.rs, types.rs, mod.rs) lives inside this repo at src/nautilus-server/src/apps/deepbook-incentives/. The provisioning script rsyncs this entire repo to EC2 at ~/nautilus — you do not need to manually clone it on EC2.

Sui CLI + Wallet

Sui CLI must be installed and configured with an active address that has SUI (for gas) and DEEP (for funding the incentive treasury). All on-chain transactions (deploy, PCR update, register, create fund) are signed by this wallet.

Phase 0: Testnet Data (DEV ONLY)

setup-testnet-data.sh

Dev/testing only — skip this in production. This script provisions a local deepbook-indexer and deepbook-server with real testnet data so you have something for the enclave to score against. Run from your machine:

./setup-testnet-data.sh              # last 24h
./setup-testnet-data.sh --hours 48   # last 48h
./setup-testnet-data.sh --skip-build # reuse binaries

What it does:

  • Builds deepbook-indexer and deepbook-server in release mode
  • Creates a deepbook_testnet PostgreSQL database (runs migrations automatically)
  • Queries testnet RPC for the latest checkpoint, calculates ~24h lookback
  • Starts the indexer with --first-checkpoint to pull from Sui's testnet remote store
  • Starts the server on :9008 with testnet package IDs

Once running, the enclave can reach http://localhost:9008/incentives/pool_data/... to get real order/fill/stake data for scoring. In production, you'd use an existing deepbook-server instance instead.

Phase 1: Deploy Contract

deploy.ts

Run from your machine. Compiles the maker_incentives Move package and publishes it to Sui. This creates two on-chain objects:

  • Package ID — the published smart contract address
  • EnclaveConfig — a shared object that will hold PCR values (currently empty)

Both IDs are saved to deployed.<network>.json, which every subsequent script reads.

Configure deepbook-indexer

After deploy. Once you have the Package ID, you must add it to the deepbook-indexer configuration so the indexer starts watching for events emitted by the maker_incentives contract. The indexer writes these events into PostgreSQL, which deepbook-server then exposes via the 3 event query endpoints:

  • FundCreated → maker_incentive_fund_created table
  • EpochResultsSubmitted → maker_incentive_epoch_results_submitted table
  • RewardClaimed → maker_incentive_reward_claimed table

Without this step, the indexer won't know about the new contract, the event tables will stay empty, and the event query endpoints will return no data.

Phase 2: Provision Enclave

setup-ec2.sh (one-time, fresh instance only)

Run from your machine. If the EC2 instance is brand new, run this first to install all dependencies:

./setup-ec2.sh --host 18.191.170.1 --key ~/.ssh/enclave.pem

Installs Docker, nitro-cli, vsock-proxy, socat, jq, make, git. Configures the Nitro allocator with 3 GiB for enclave VMs. Adds ec2-user to the docker group. Only needed once per EC2 instance — after this, disconnect and reconnect SSH for group changes to take effect.

provision-enclave.sh

Run from your machine. This single script handles the entire EC2-side setup via SSH. Usage:

./provision-enclave.sh --host 18.191.170.1 --key ~/.ssh/enclave.pem [--debug]

What it does, step by step:

  1. Rsync the Nautilus repo -- copies your local ~/projects/nautilus to ec2-user@HOST:~/nautilus/. This includes the Makefile, Containerfile, run.sh, and your app code. One command, no manual cloning needed.
  2. Build the EIF -- runs make ENCLAVE_APP=deepbook-incentives on EC2 inside Docker. This compiles the Rust scoring code into an EIF (Enclave Image File), a bootable VM image. The build also produces out/nitro.pcrs with three SHA-384 hashes that uniquely identify this exact build.
  3. Start the enclave -- runs nitro-cli run-enclave which boots the EIF as an isolated Nitro Enclave VM with no network access.
  4. Set up networking -- since the enclave has no network, the script creates two bridges:
    • socat TCP4-LISTEN:3000 → VSOCK-CONNECT:CID:3000 — inbound: lets external callers (like submit-epoch.ts) reach the enclave's HTTP server through the host's port 3000
    • vsock-proxy 9008 127.0.0.1 9008 — outbound: lets the enclave call deepbook-server on the host's port 9008. The enclave sees this as VSOCK port 9008, which vsock-proxy forwards to localhost:9008
    It also sends an empty {} secrets payload via VSOCK port 7777 — Nautilus's run.sh blocks until it receives this, so the enclave server doesn't start until secrets are delivered.
  5. Health check -- retries curl http://HOST:3000/health_check up to 5 times. On success, prints the enclave's Ed25519 public key and the PCR hashes. These are needed for the next phase.
Phase 3: Register Enclave On-Chain

update-pcrs.ts

Run from your machine. Takes the three PCR hashes from out/nitro.pcrs and writes them into the on-chain EnclaveConfig. From this point forward, the contract will only accept signatures from an enclave whose attestation matches these exact hashes. In dev, use --debug to set all zeros (bypasses verification).

register-enclave.ts

Run from your machine. Calls /get_attestation on the running enclave. This returns a Nitro Attestation Document -- a cryptographic certificate from AWS that says "this enclave is running code with these PCR values, and here is its Ed25519 public key." The script submits this document on-chain, where the contract verifies the PCRs match EnclaveConfig, extracts the public key, and stores it in the Enclave object. All future submit_epoch_results calls must be signed by this key.

Phase 4: Create Fund + Automation

create-fund.ts

Run from your machine. Creates an IncentiveFund on Sui targeting a specific DeepBook trading pool. You specify the scoring parameters (alpha, reward per epoch, timing) and deposit DEEP tokens as the reward treasury. The script returns a FundOwnerCap to your wallet and appends the fund ID to deployed.<network>.json so the automation scripts know about it.

setup-systemd.sh

Run on the EC2 host. Installs a systemd timer + oneshot service. The timer fires daily at 00:15 UTC, which launches submit-all-epochs.ts. That script reads deployed.<network>.json, loops through every fund, calls the enclave for scores, and submits on-chain. After this step, the system is fully autonomous.

Output

deployed.<network>.json

The local config file that accumulates all important object IDs as you run each provisioning step:

  • packageId -- the published Move package address
  • enclaveConfigId -- the EnclaveConfig shared object
  • enclave.objectId -- the registered Enclave object
  • enclave.publicKey -- the enclave's Ed25519 public key
  • funds -- map of fund_id to { poolId, ownerCapId }

This file is what submit-all-epochs.ts reads to know which funds to process. If you create additional funds later, they get appended here automatically.

When you change the scoring logic, you need to get that new code into the enclave and update the on-chain identity. The upgrade-enclave.sh script does this entire pipeline in one command. Every code change produces new PCR hashes, so the on-chain config must be updated too.

Why is the upgrade process needed?

Code Lives in the Nautilus Repo

The canonical scoring code (scoring.rs, types.rs) lives in the local Nautilus repo clone at /home/ubuntu/projects/nautilus/src/nautilus-server/src/apps/deepbook-incentives/. The standalone crate at crates/incentives/ symlinks to these files, so both the local dry-run binary and the Nautilus enclave build share the same source. No import path fixups needed.

Rsync, Not SCP

Following the official Nautilus workflow, the upgrade script rsyncs the entire Nautilus directory to EC2 in one command. This ensures the app code, Makefile, run.sh, and all framework files stay in sync. No per-file copying, no sed hacks.

Why Update PCRs After Rebuild?

Every time you rebuild the EIF, the code inside it changes, which means the SHA-384 hashes (PCRs) change. The on-chain EnclaveConfig stores the expected PCRs. If you start a new enclave but don't update the PCRs on-chain, the contract will reject its attestation because the hashes won't match. Same reason you also need to re-register -- the new enclave generates a new Ed25519 key pair on boot.

upgrade-enclave.sh

Usage

One command does everything:

  • ./upgrade-enclave.sh --host 18.191.170.1 --key ~/.ssh/key.pem --network mainnet
  • Add --debug for dev (zero PCRs, console output enabled)

What It Does (in order)

  • Rsync code -- rsync entire Nautilus repo to EC2
  • Stop old enclave -- nitro-cli terminate-enclave
  • Rebuild EIF -- make ENCLAVE_APP=deepbook-incentives (compiles new code)
  • Start new enclave -- nitro-cli run-enclave with the fresh EIF
  • Restart networking -- kill old socat/vsock-proxy, start new ones
  • Health check -- curl /health_check to verify enclave is alive
  • Update PCRs -- write new hashes to on-chain EnclaveConfig
  • Re-register -- fetch new attestation, register new public key on-chain

After Upgrade

No changes needed to funds, automation, or config files. The next time submit-all-epochs.ts runs, it will call the enclave which now runs the updated scoring logic. The contract will verify the new signature against the newly registered public key. Everything else stays the same.
